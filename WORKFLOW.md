## 1. Project Overview

CampusFlow is a multi-tenant AI operating system for engineering college students. It combines a structured relational database of campus data with a large language model (Google Gemini) to deliver a conversational assistant — BLACKY — that answers deeply personalized academic questions grounded in each student's real records.

The system is not a generic chatbot. It is a **retrieval-augmented assistant**: before every AI call, the backend programmatically fetches the logged-in student's data from SQLite and injects it as structured context into the model prompt. The model never guesses; it reasons over real numbers.

**Tech stack**

| Layer | Technology |
|---|---|
| UI framework | Streamlit (Python 3.12) |
| Database | SQLite via `campus.db` |
| AI model | Google Gemini 1.5 Flash (`google-generativeai` SDK) |
| Password hashing | bcrypt (work factor 12) |
| Environment config | python-dotenv (`.env` file) |
| Theme | Custom CSS injected via `st.markdown`, base theme via `.streamlit/config.toml` |

## 2. File Structure & Module Responsibilities

```
campusflow/
├── app.py              # UI orchestrator — renders all pages after auth
├── auth.py             # Login and Register UI pages + session gate
├── db.py               # All database logic: connection, queries, context builder
├── init_db.py          # One-time schema creation + mock data seeding
├── campus.db           # SQLite database (auto-generated by init_db.py)
├── .env                # Secret credentials (GEMINI_API_KEY)
├── .gitignore          # Excludes .env, .venv, campus.db from version control
├── .streamlit/
│   └── config.toml     # Streamlit theme: dark base, black/white palette
├── requirements.txt    # Pinned Python dependencies
├── README.md           # Hackathon pitch document
└── WORKFLOW.md         # This document
```

### Module contract summary

**`db.py`** — Single source of truth for all data access. No SQL runs anywhere else in the codebase. Exposes typed functions for every query. Also owns the `build_llm_context()` function that assembles the AI prompt payload.

**`auth.py`** — Owns the authentication gate. Exports one function: `render_auth_page()`. If `st.session_state.user` is already set, it returns immediately (no-op). If not, it renders login/register and calls `st.stop()` to block the rest of `app.py` from executing.

**`app.py`** — Assumes the user is authenticated (guaranteed by auth gate). Reads `st.session_state.user` to scope all queries. Renders the sidebar, header, status cards, tabbed panels, and chatbot. Owns the `query_blacky()` function that calls the Gemini API.

**`init_db.py`** — Standalone script. Drops and recreates all tables, seeds three demo users with realistic but varied academic data. Run once before launching the app.

## 3. Application Boot Sequence

Every time Streamlit reruns `app.py` (on page load, widget interaction, or `st.rerun()`), the following sequence executes top to bottom:

```
1. load_dotenv()                    — Load GEMINI_API_KEY from .env into os.environ
2. st.set_page_config()             — Set page title, layout, sidebar state
3. st.markdown(CSS)                 — Inject global black/white stylesheet
4. render_auth_page()               — Auth gate check (see section 4)
   └── if not authenticated → render login/register → st.stop()
   └── if authenticated     → return (continue execution)
5. user = st.session_state.user     — Bind current user dict
6. Sidebar renders                  — API key input, profile card, leave form, sign out
7. Compute: now, day_name, time_str — Live datetime context
8. Header renders                   — BLACKY bar + calendar badge
9. Attendance alert check           — If any subject < 75%, render alert card
10. Status cards render             — Next class, mess, attendance, nearest deadline
11. Tabs render                     — Schedule, Attendance, Marks, Leaves, Events
12. Chat renders                    — History replayed from st.session_state.messages
13. st.chat_input()                 — Waits for user input (triggers rerun on submit)
```

Steps 1–4 execute even before any user-specific data is loaded. Steps 5–13 only execute if the user is authenticated.

## 4. Authentication Flow

### Login path

```
User submits form (username + password)
    │
    ▼
auth.py: login_user(username, password)
    │
    ▼
db.py: SELECT * FROM users WHERE username=? (case-insensitive)
    │
    ├── Row not found → return None → st.error("Invalid credentials")
    │
    └── Row found → bcrypt.checkpw(password, stored_hash)
                        │
                        ├── False → return None → st.error()
                        │
                        └── True → return user dict
                                       │
                                       ▼
                               st.session_state.user = user dict
                               st.session_state.messages = []
                               st.rerun()  →  auth gate passes, dashboard loads
```

### Password security

Passwords are hashed with `bcrypt` at work factor 12, which produces a cost of approximately 300ms per hash on commodity hardware. This is intentionally slow to resist brute-force attacks. The hash is stored in the `password_hash` column of the `users` table. The plaintext password is never persisted anywhere.

Verification uses `bcrypt.checkpw()`, which is constant-time and safe against timing attacks.

### Register path

```
User submits register form
    │
    ├── Client-side checks: all fields filled, password ≥ 6 chars, passwords match
    │
    ▼
db.py: SELECT id FROM users WHERE username=? 
    │
    ├── Exists → return (False, "Username already exists")
    │
    └── Not exists → bcrypt.hashpw(password, gensalt(12))
                         │
                         ▼
                     INSERT INTO users (username, password_hash, name, dept, sem)
                     return (True, "Account created successfully")
```

New users get no attendance/marks records on registration. BLACKY handles this gracefully — the context builder emits empty sections and the system prompt instructs BLACKY to guide the user rather than crash.

### Session management

`st.session_state` is Streamlit's in-memory session store, scoped per browser tab connection. It persists across reruns within a session but is cleared when the tab closes or the user signs out. Sign-out sets `st.session_state.user = None` and calls `st.rerun()`, which causes the auth gate to render again.

## 5. Database Architecture

### Schema overview

```
users                           (id, username, password_hash, name, department, semester)
    │
    ├── attendance_records      (id, user_id FK, subject_name, total_classes_conducted, classes_attended)
    ├── academic_marks          (id, user_id FK, subject_name, exam_type, marks_obtained, total_marks)
    └── leave_applications      (id, user_id FK, leave_date, reason, status)

timetables                      (id, department, day, time_slot, subject, room)
mess_menu                       (id, day, meal_type, time_window, menu_items)
college_events                  (id, event_name, event_type, due_date, description, venue, organizer)
college_calendar                (date PK, day_type, description)
```

Tables are divided into two categories:

**Per-user tables** — `attendance_records`, `academic_marks`, `leave_applications`. All rows carry a `user_id` foreign key. Every query against these tables passes `WHERE user_id=?` with the logged-in user's ID. There is no way for one user's data to appear in another user's session.

**Shared tables** — `timetables`, `mess_menu`, `college_events`, `college_calendar`. These are institution-wide datasets. Queries filter by `department` (from the user's profile) or `day` (from live datetime), not by user ID.

### Key computed values

`db.py` computes several derived values that are not stored in the database:

- `attendance.percentage` = `classes_attended / total_classes_conducted * 100`, rounded to 1 decimal
- `marks.percentage` = `marks_obtained / total_marks * 100`, rounded to 1 decimal
- `get_overall_attendance()` = weighted average across all subjects (total attended / total conducted)
- `get_average_internal_marks()` = average percentage across all Internal-1 and Internal-2 rows for the user
- `get_next_class()` = scans today's timetable in time order and returns the first slot whose start time >= current time

## 6. Dashboard Rendering Pipeline

When `app.py` renders the main area, it executes these DB calls in order:

```python
att_records = db.get_attendance(user["id"])          # all subject attendance rows
overall_att = db.get_overall_attendance(user["id"])  # single computed float
low_att     = db.get_low_attendance(user["id"])      # subjects below 75%
avg_int     = db.get_average_internal_marks(user["id"])

next_class      = db.get_next_class(user["department"], day_name, time_str)
current_meal    = db.get_current_meal(day_name, time_str)
upcoming_events = db.get_upcoming_events(3)
cal_entry       = db.get_calendar_entry(today_iso)
```

These results are computed once per page render and reused across the status cards, alert banner, tabs, and sidebar profile card. No data is fetched twice.

### Attendance alert logic

```python
if low_att:                 # list is non-empty
    render red alert card   # lists every subject name + percentage + counts
```

The alert card only renders if at least one subject is below 75%. It is not a warning — it is a hard visual interrupt, styled with a red left border and dark red background, positioned directly above the status cards so it is impossible to miss.

### Calendar badge logic

```python
cal_entry = db.get_calendar_entry(today_iso)
if cal_entry:               # only render if today is in the calendar table
    render badge            # Working Day (green) / Official Holiday (red) / Weekend (grey)
```

The badge is rendered as a separate `st.markdown` call after the header bar, not inside the header's f-string. This prevents empty HTML tags from appearing when no calendar entry exists for the current date.

## 7. How the AI Works — Full Orchestration

BLACKY is not a fine-tuned model. It is a **retrieval-augmented generation (RAG) pattern** implemented manually without a vector database. The architecture:

```
User types message
      │
      ▼
1. INTENT DETECTION (keyword matching in db.build_llm_context)
      │
      ▼
2. SELECTIVE DB FETCHES (only relevant tables queried)
      │
      ▼
3. CONTEXT ASSEMBLY (structured plaintext block built from query results)
      │
      ▼
4. PROMPT CONSTRUCTION (system instruction + context block + user question)
      │
      ▼
5. GEMINI API CALL (stateful multi-turn chat)
      │
      ▼
6. RESPONSE RETURNED and appended to st.session_state.messages
      │
      ▼
7. st.rerun() — page rerenders with new message in chat history
```

### Why this is not a naive LLM call

A naive implementation would send: `"system: you are a campus assistant"` + `"user: what's my attendance?"` and hope the model knows the answer. That produces hallucinations.

CampusFlow's approach: before the model sees the question, the backend runs real SQL queries and packs the actual numbers into the prompt. The model's job is **reasoning and synthesis**, not recall.

Example of what the model actually receives for the question *"Am I safe to take a leave this Friday?"*:

```
DATABASE CONTEXT:
---
STUDENT PROFILE:
  Name: Alex Johnson
  Department: CSE | Semester: 5
  Current Day: Sunday | Time: 03:45 PM

ATTENDANCE RECORDS (Overall: 77.3%):
  Compiler Design: 30/38 (78.9%)
  Computer Networks: 25/36 (69.4%)
  Data Structures & Algorithms: 38/40 (95.0%)
  Database Management Systems: 40/42 (95.2%)
  Machine Learning: 35/40 (87.5%)
  Operating Systems: 26/38 (68.4%)
  [WARNING] Subjects below 75%: Computer Networks (69.4%), Operating Systems (68.4%)

LEAVE HISTORY:
  2025-06-10 — Fever and cold — medical leave [Approved]
  2025-06-18 — Family function out of city [Approved]
  2025-06-25 — Doctor appointment — follow-up [Pending]

UPCOMING EVENTS / DEADLINES:
  [Assignment] DSA Assignment Submission — 2025-06-25: Submit DSA assignment...
  [Exam] Mid-Semester Examination (CSE) — 2025-07-14: Mid-semester exams...
---

Question: Am I safe to take a leave this Friday?
```

The model receives concrete numbers, not vague descriptions. It can calculate exactly how many more classes Alex can miss in Operating Systems before falling below the academic cutoff.

## 8. The Context Builder — Core of the AI Engine

`db.build_llm_context()` in `db.py` is the most important function in the system. It decides what data to fetch and how to structure it.

### Always-included sections

Regardless of what the user asked, these are always included:

- Student profile (name, department, semester, current day and time)
- Full attendance records with computed percentages and a warning block if any subject is below 75%
- Full academic marks with computed percentages and average internals
- Full leave history

These are always included because they are the "working memory" that makes BLACKY a personalized advisor rather than a generic assistant. Even if the user asks about lunch, BLACKY knows in the background that their OS attendance is 68%, and can weave that into an advisory note if relevant.

### Conditionally-included sections

These are only fetched when keyword matching detects relevance in the user's message:

| Keywords detected | Section added |
|---|---|
| class, lecture, schedule, timetable, subject, room, lab, next | Today's department timetable |
| eat, food, menu, mess, breakfast, lunch, dinner, snack, meal | Mess menu (specific meal or current) |
| event, hackathon, deadline, exam, assignment, workshop, seminar, submit, safe, leave, upcoming | Upcoming events or department exams |

This selective fetching keeps the context window focused. Sending the full mess menu, full timetable, and all 19 events on every single query would waste tokens and dilute the model's attention on what actually matters.

### Context assembly order

```python
parts = []
parts.append(STUDENT_PROFILE)
parts.append(ATTENDANCE_RECORDS)   # always
parts.append(ACADEMIC_MARKS)       # always
parts.append(LEAVE_HISTORY)        # always
if schedule_keywords: parts.append(TIMETABLE)
if food_keywords:     parts.append(MESS_MENU)
if event_keywords:    parts.append(EVENTS)
if calendar_entry:    parts.append(CALENDAR)
return "\n\n".join(parts)
```

### System prompt design

The system instruction sent to Gemini defines BLACKY's identity and behavioral constraints:

```
You are BLACKY, the AI assistant embedded in CampusFlow.
You have COMPLETE ACCESS to the student's personal academic data:
- Subject-wise attendance counts and percentages
- Internal exam marks across all subjects
- Leave history (dates, reasons, approval status)
- Today's class schedule, mess menu, upcoming events and deadlines

Rules:
- Answer using ONLY the DATABASE CONTEXT provided. Never fabricate data.
- For attendance safety questions, calculate exactly:
  classes_needed = ceil(0.75 * (total + future) - attended)
- Format multi-item answers as short bullet lists.
- If no records exist yet, gracefully guide the student.
- Address the student by first name.
```

The attendance safety formula is explicitly included in the system prompt because it is a specific calculation the model must perform correctly for the most common student query type.

### Multi-turn conversation

BLACKY maintains conversation context across turns. Before each API call, the full `st.session_state.messages` history (excluding the current message) is converted to Gemini's `{"role": "user"/"model", "parts": [...]}` format and passed as `history` to `model.start_chat()`.

This means BLACKY remembers: *"You told me you were planning to skip Friday. Given that, your Computer Networks attendance would drop to 67.2%, which is 3 more classes below the cutoff."*

## 9. Request Lifecycle — End to End

Here is the complete journey of a single user message from keypress to rendered response:

```
[1] User types in st.chat_input()
        Streamlit captures input → triggers script rerun

[2] app.py appends user message to st.session_state.messages

[3] api_key check
        └── No key → reply = static offline message → skip to [7]
        └── Key exists → continue

[4] db.build_llm_context(prompt, user, day_name, time_str)
        ├── Fetches attendance_records WHERE user_id=?
        ├── Fetches academic_marks WHERE user_id=?
        ├── Fetches leave_applications WHERE user_id=?
        ├── (conditional) Fetches timetables WHERE department=? AND day=?
        ├── (conditional) Fetches mess_menu WHERE day=? [AND meal_type=?]
        ├── (conditional) Fetches college_events or college_calendar
        └── Returns structured plaintext context string

[5] query_blacky(api_key, prompt, context, history)
        ├── genai.configure(api_key=api_key)
        ├── GenerativeModel("gemini-1.5-flash", system_instruction=...)
        ├── Converts message history to Gemini format
        ├── model.start_chat(history=gemini_history)
        ├── chat.send_message("DATABASE CONTEXT:\n...\n\nQuestion: " + prompt)
        └── Returns response.text

[6] app.py appends BLACKY's reply to st.session_state.messages

[7] st.rerun()
        └── Page rerenders
        └── Chat history loop replays all messages including new ones
        └── New user + BLACKY message pair appears in chat
```

Total DB queries per chat message: typically 3–6 (always: attendance + marks + leaves; conditionally: timetable, mess, events). All queries are parameterized — no string interpolation in SQL, no SQL injection surface.

## 10. Multi-Tenancy & Data Isolation

The system is multi-tenant by design. Tenant isolation is enforced at the query layer in `db.py`, not in the UI.

Every user-scoped query is structured as:

```sql
SELECT * FROM attendance_records WHERE user_id = ?
SELECT * FROM academic_marks WHERE user_id = ?
SELECT * FROM leave_applications WHERE user_id = ?
```

The `?` is always bound to `st.session_state.user["id"]`, which is set only by `login_user()` after successful bcrypt verification. There is no way to construct a query that leaks another user's data — the `user_id` is never taken from user input, only from the server-side session.

Shared data (timetables, mess menu, events, calendar) is filtered by `department` and `day`, which also come from the session — not from URL parameters or user-controlled input.

## 11. Current Limitations

These are known constraints of the current prototype that will be addressed in future phases:

| Limitation | Impact | Mitigation in place |
|---|---|---|
| SQLite is a file-level database with no concurrent write safety | Multiple simultaneous users editing data could cause write conflicts | `check_same_thread=False` allows read access; writes are low-frequency (only leave applications) |
| Attendance and marks are seeded mock data | New registered users have no records; BLACKY cannot give attendance advice | Graceful empty-state handling in context builder and system prompt |
| Session state clears on browser close | User must log in again each visit | By design for the prototype; persistent sessions via server-side tokens planned |
| API key stored in browser session | Key is not persisted securely beyond the session | `.env` file support provided as the recommended alternative |
| Gemini API latency | Each chat response takes 1–3 seconds | `st.spinner()` provides feedback; streaming responses are a planned upgrade |
| No attendance update UI | Students cannot update their own attendance counts | Admin panel and bulk import planned in Phase 3 |
| Calendar only covers June–November 2025 | Queries outside this range return no calendar entry | Expanding to full academic year in next seed update |

## 12. Future Implementation Roadmap

### Phase 3 — Data Management & Admin Layer

- **Student self-service**: UI for students to update their attendance counts and add new marks records as their semester progresses
- **Admin panel**: A separate admin login (role-based access) for faculty to bulk-update attendance, enter marks, and approve/reject leave applications
- **Bulk data import**: CSV upload for timetable changes, mark entry, and attendance sheets
- **Notifications**: Scheduled checks that fire an alert (email or in-app) when a student's attendance drops below 75% in any subject

### Phase 4 — Production Infrastructure

- **PostgreSQL migration**: Replace SQLite with PostgreSQL for concurrent multi-user writes. The `db.py` abstraction layer means only the connection string and a few query patterns need to change
- **Containerization**: Dockerfile + docker-compose for reproducible deployments
- **AWS deployment**:
  - App hosted on ECS (Fargate) or Elastic Beanstalk
  - Database on RDS (PostgreSQL)
  - Secrets managed via AWS Secrets Manager (replaces `.env`)
  - Static assets on S3 + CloudFront
  - Application load balancer for horizontal scaling

### Phase 5 — AI Capability Expansion

- **Streaming responses**: Gemini supports streaming; implementing `response.stream()` and `st.write_stream()` would eliminate the spinner and make BLACKY feel instantaneous

- **Proactive alerts**: A background scheduler (APScheduler or AWS EventBridge) checks each user's attendance daily and pushes a personalized BLACKY message: *"Alex, you have 3 classes in Operating Systems this week. Skipping any of them will drop your attendance to 62.1%. Recommend attending all three."*

- **GPA prediction model**: A lightweight regression model (scikit-learn, trained on historical mark patterns) that answers *"What do I need to score in the End-Semester exam to achieve a GPA of 8.5?"* with a calculated target score rather than a generic estimate

- **Semantic search over context**: Replace keyword matching in `build_llm_context()` with an embedding-based retrieval step using `text-embedding-004` (Gemini). Student query → query embedding → cosine similarity against pre-embedded chunks of campus data → top-k chunks sent to the model. This handles paraphrases, typos, and complex multi-intent queries more robustly than keyword lists

- **Tool use / function calling**: Expose database functions as Gemini tool declarations. The model decides which tools to call based on the query, rather than the context builder deciding by keywords. This is the cleanest long-term architecture for the AI layer

- **Memory across sessions**: Store conversation summaries in a `chat_sessions` table. On login, retrieve the last session summary and inject it as a "memory" block in the system prompt. BLACKY will remember that you discussed exam prep last week

### Phase 6 — Platform Expansion

- **Mobile PWA**: Streamlit's responsive layout supports mobile browsers. Adding a PWA manifest and service worker would enable home screen installation
- **REST API layer**: FastAPI backend exposing the same `db.py` functions as authenticated JSON endpoints, enabling native mobile apps or third-party integrations
- **ERP integration**: Replace mock data with a live feed from the college ERP system via scheduled sync jobs, making attendance and marks data real-time accurate
- **Multi-college SaaS**: Introduce a `college_id` tenant key across all shared tables, allowing multiple institutions to share one deployment with fully isolated data
